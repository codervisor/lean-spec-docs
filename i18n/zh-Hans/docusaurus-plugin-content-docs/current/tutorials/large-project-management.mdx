# 大型项目与多功能管理

当项目超出少数几个规格说明时，您需要更好的工具来保持可见性和协调工作。学习在大规模工作中的关键项目管理命令。

**时间**：15 分钟  
**成果**：掌握大型代码库的项目管理工具

:::tip 适用于高级用户
本教程假设您熟悉基本的 LeanSpec 工作流程。如果您是新手，请从 [编写第一个规格说明](./first-spec-with-ai) 开始。
:::

## 关键 PM 工具

LeanSpec 提供五个关键工具来管理大型项目：

1. **`lean-spec board`** - 项目状态的看板视图
2. **`lean-spec deps <spec>`** - 依赖关系分析
3. **`lean-spec stats`** - 项目健康指标
4. **`lean-spec list --tags <tags>`** - 按团队/功能过滤
5. **`lean-spec search <query>`** - 跨规格说明的语义搜索

让我们探索每一个。

## 1. 看板视图 - 项目概览

看板为您提供整个项目的即时可见性：

```bash
lean-spec board
```

按状态列显示所有规格说明：**计划中** → **进行中** → **完成** → **已归档**

**来自 LeanSpec 项目的示例：**
- 计划中：15 个规格说明（新功能、增强）
- 进行中：0 个规格说明（当前没有活跃工作）
- 完成：64 个规格说明（已发布功能）
- 已归档：0 个规格说明

看板还显示项目健康指标：
- 完成率
- 被阻塞的规格说明数量（等待依赖项）
- 最近活动

**使用场景：** 开始一天的工作、冲刺规划、站会

**专业提示：** 具有未满足 `depends_on` 关系的规格说明会被标记为 "被阻塞"

## 2. 依赖关系分析

在进行更改之前了解规格说明之间的关系：

```bash
lean-spec deps <spec> --mode complete
```

显示三种类型的关系：
- **依赖项** (`depends_on`)：此规格说明在开始之前需要什么
- **被依赖项** (`required_by`)：哪些规格说明需要这个
- **相关项**：上下文相关的规格说明

**来自 LeanSpec 的真实示例：**

```bash
lean-spec deps 114  # 示例项目脚手架
```

显示：
- **相关于**：113（教程简化）
- 两个规格说明共同改善教程体验

### 影响分析

在重构之前，检查影响范围：

```bash
lean-spec deps <spec> --mode impact
```

显示：
- 直接依赖项（直接依赖此规格说明的规格说明）
- 传递依赖项（下游受影响的规格说明）
- 总影响数

**使用场景：** 规划重构、破坏性更改、架构决策

**专业提示：** `--mode upstream` 仅显示依赖项，`--mode downstream` 仅显示被依赖项

## 3. 项目统计

获取整个项目的健康指标：

```bash
lean-spec stats
```

显示：
- **状态分布**：每个状态中有多少规格说明
- **优先级分布**：关键工作与可选工作
- **热门标签**：最常见的类别
- **最近活动**：最新的规格说明更新

**使用场景：** 冲刺回顾、季度规划、跟踪速度

**注意这些危险信号：**
- 太多 "计划中" 规格说明（没有交付）
- 太多 "进行中" 规格说明（工作未完成）
- 无最近活动（项目停滞）

## 4. 过滤和组织

### 按标签

按团队、功能领域或技术组织规格说明：

```bash
# 查看团队的所有规格说明
lean-spec list --tags team-frontend

# 查看功能领域的所有规格说明
lean-spec list --tags documentation

# 查看里程碑的所有规格说明
lean-spec list --tags v0.3.0
```

**真实的 LeanSpec 标签：**
- `web`、`cli`、`mcp`、`docs` - 组件领域
- `ux`、`dx`、`performance` - 质量属性
- `v0.2.0`、`v0.3.0` - 发布里程碑

### 按优先级

专注于关键工作：

```bash
lean-spec list --priority critical
lean-spec list --priority high
```

### 按状态

跟踪特定状态的工作：

```bash
lean-spec list --status planned      # 准备开始
lean-spec list --status in-progress  # 当前活跃
lean-spec list --status complete     # 已发布
```

## 5. 语义搜索

按内容而不仅仅是名称查找规格说明：

```bash
lean-spec search "authentication"
lean-spec search "performance optimization"
lean-spec search "dependency graph"
```

搜索使用语义匹配——它理解上下文，即使没有精确的关键词也能找到相关的规格说明。

**使用场景：** 查找相似模式、发现相关工作、发现现有解决方案

## 组织模式

### 编号约定

LeanSpec 使用数字范围来分组相关工作：

- **010-029**：项目设置和工具
- **040-059**：核心功能和工作流  
- **060-079**：高级功能和集成
- **080-099**：基础设施和架构
- **100+**：持续改进和完善

**好处：**
- 一眼看出功能领域
- 轻松在逻辑位置插入新规格说明
- 在领域内保持时间顺序

### 标签策略

有效的标签跨多个维度组织工作：

```bash
# 组件标签
web, cli, mcp, docs

# 质量属性标签
ux, dx, performance, security

# 发布里程碑标签
v0.2.0, v0.3.0, v0.4.0

# 团队/所有权标签（如果是多团队）
team-frontend, team-backend, team-infra
```

**专业提示：** 在整个项目中保持标签一致。在 README 中记录您的标签约定。

## 使用依赖关系

### 创建依赖关系

当一个规格说明阻塞另一个时链接规格说明：

```bash
# 规格说明 B 依赖于规格说明 A 首先完成
lean-spec link B --depends-on A

# 规格说明 C 依赖于 A 和 B
lean-spec link C --depends-on A,B
```

### 创建软引用

链接相关规格说明而不阻塞：

```bash
# 共享上下文但不相互阻塞的规格说明
lean-spec link A --related B,C
```

**关键区别：** `depends_on` 阻塞工作（硬依赖），`related` 提供上下文（软引用）

### 移除链接

清理过时的关系：

```bash
lean-spec unlink A --depends-on B    # 移除依赖关系
lean-spec unlink A --related C       # 移除软引用
```

## 成功技巧

### 从简单开始

不要从第一天就过度组织：
1. 从基本状态跟踪开始
2. 当需要集中工作时添加优先级
3. 当需要更好的过滤时添加标签
4. 当工作开始阻塞时添加依赖关系

### 定期维护

保持规格说明健康：
- **每天：** 更新活跃工作的状态
- **每周：** 在站会/规划中回顾看板
- **每月：** 检查停滞的规格说明，更新优先级
- **每季度：** 归档完成的工作，审核标签

### 沟通模式

使用规格说明进行异步协调：
- 在开始工作之前检查 `lean-spec deps`
- 在站会中运行 `lean-spec board`
- 在冲刺回顾中分享 `lean-spec stats`
- 在提问之前使用 `lean-spec search` 查找上下文
