# 规范

**规范**是一份轻量级的对齐文件，用来捕获工作单元的意图、约束和成功标准。它同时服务于人类与 AI 代理，确保所有执行者都知道下一步该做什么以及为什么这么做。

## 为什么重要
- 在交接和 AI 会话之间提供持久记忆
- 在实现开始前迫使大家对问题和约束达成一致
- 通过明确的成功标准和护栏让自主执行更安全

## 核心结构
1. **前置元数据** —— 状态、优先级、标签、依赖等机器可读字段
2. **概述** —— 问题、为什么现在处理、预期结果
3. **计划/设计** —— 阶段、权衡、方法或子规范链接
4. **验证** —— 如何证明工作真正完成

将 README 控制在约 2,000 令牌内。一旦超出工作记忆，就把细节拆进子规范并从计划部分链接出去。

## 何时使用子规范
子规范是位于规范目录中的额外 Markdown 文件（如 `DESIGN.md`、`IMPLEMENTATION.md`、`TESTING.md`）。它们保留上下文经济，同时满足不同受众的深度需求。

| 何时拆分 | 示例 |
| --- | --- |
| README 逼近 2,000 令牌 | 架构图、多阶段上线计划 |
| 不同受众需要不同深度 | README 提供结论，`IMPLEMENTATION.md` 提供任务清单 |
| 需要专用工件 | `TESTING.md`、迁移 Runbook、回滚计划 |

```
specs/082-web-sync/
	README.md          # 概览 + 决策背景
	DESIGN.md          # 架构图与流程
	IMPLEMENTATION.md  # 阶段或任务列表
	TESTING.md         # 验证细节
```

## 快速示例
```markdown
# 042-api-rate-limiting
实现令牌桶速率限制（每个 API 密钥 100 次/分钟）
以保护 API 免受 2% 重度用户的滥用。
```
