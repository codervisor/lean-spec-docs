# 理解规范

本页重点介绍如何在了解规范存在的原因后，阅读、演变和维护 LeanSpec 文档。

## 结构回顾
1. **前置元数据** — 状态、标签、优先级、日期。
2. **概述** — 问题、为什么现在做、期望结果。
3. **设计/计划** — 方法、阶段、权衡。
4. **测试/验证** — 可测量的成功标准。
5. **注释/链接** — 相关规范或研究的引用。

保持每个部分有目的性。如果细节让 README 不堪重负，将它们提取到子规范中。

## 阅读与编写
- **阅读**：扫描前置元数据 → 概述 → 成功标准。决定你是否有足够的上下文来采取行动。
- **编写**：从问题 + 为什么现在开始，然后捕获约束和完成定义。只有在有助于执行时才添加阶段。
- **编辑**：当现实发生变化时更新规范。像对待代码一样对待它——小的频繁提交胜过巨大的重写。

## 保持规范活跃
- 工作开始或停止后立即更新状态。
- 在 `## 注释` 或 `## 演变` 部分下添加发现或范围变更。
- 引用相关规范，以便未来读者可以跟踪线索。
- 运行 `lean-spec validate` 以捕获缺失的元数据或超大文件。

## 规范演变检查清单
- ☐ 仍然低于约 2,000 令牌（或拆分为子规范）
- ☐ 问题陈述与当前理解相符
- ☐ 成功标准仍然可测试
- ☐ 链接指向当前文件（示例、仪表板、MCP 命令）
- ☐ 依赖关系反映真实的阻碍因素

## 与 AI 代理协作
1. 共享规范路径或使用 MCP `search_specs` 工具，以便代理可以加载上下文。
2. 让代理向你总结规范——确保对齐。
3. 让代理起草代码或文档；根据结果或偏差更新规范。
4. 实现后，确认成功标准并将规范标记为完成。
