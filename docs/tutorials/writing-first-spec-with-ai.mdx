## Creating the Spec

In your AI tool, describe what you want:

```
Create a new spec for adding email notifications to password resets.
Users should receive a confirmation email with:
- Reset confirmation message
- Security tips
- Link to sign in

Use the existing email service. Keep it simple.
```

The AI reads your project structure and creates a spec in `specs/014-reset-email-notifications/`. Here's what it generates:

```markdown title="specs/014-reset-email-notifications/README.md"
---
status: planned
created: '2025-11-24'
priority: medium
---
```

## Implementation

Now ask the AI to implement it:

```
Implement spec 014.
```

The AI will:
- Read the spec to understand requirements
- Create the email template file
- Add email sending to `reset-password.ts`
- Write tests

## What Just Happened

You experienced spec-driven development with AI:

**Without LeanSpec:**
- Jump into code, forget requirements
- Lose context switching between files
- Hard for others to understand why
- No record of what was delivered

**With LeanSpec:**
- Clear intent captured upfront
- AI implements with full context
- Documented decisions for future you
- Complete audit trail of what shipped

The spec becomes your **shared memory** between you, the AI, and your team. Everyone knows what was built and why.
