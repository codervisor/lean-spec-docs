# Working with Teams

**Time**: 15 minutes  
**Goal**: Learn to use LeanSpec for team collaboration - assigning work, coordinating, and reviewing specs

## What You'll Learn

- Assign specs to team members
- Use priorities for team planning
- Review others' specs effectively
- Coordinate work across multiple people
- Handle spec handoffs

## Prerequisites

- Completed previous tutorials ([Your First Spec](./your-first-spec), [SDD Workflow](./sdd-workflow-feature-development), [Managing Multiple Specs](./managing-multiple-specs))
- LeanSpec installed and initialized project
- Understanding of basic spec operations

## The Scenario

You're on a team of 3 developers building a social media app:
- **Alice** (you): Frontend developer
- **Bob**: Backend developer
- **Carol**: Full-stack developer

You need to coordinate work on the news feed feature, which involves multiple specs.

## Step 1: Create Team Specs

Create specs for different parts of the news feed:

```bash
lean-spec create news-feed-api
lean-spec create news-feed-ui
lean-spec create post-reactions
lean-spec create comment-system
```

Assign specs based on expertise:

```bash

lean-spec update 007 --assignee "Bob"

lean-spec update 009 --assignee "Carol"
```

## Step 3: Set Up Dependencies

The UI depends on the API:

```bash
lean-spec link 008 --depends-on 007
```

Reactions and comments are related but independent:

```bash
lean-spec link 009 --related 010
```

## Step 4: Prioritize as a Team

Set priorities for the sprint:

```bash
lean-spec update 007 --priority high --tags "backend,api,sprint-1"
lean-spec update 008 --priority high --tags "frontend,ui,sprint-1"
lean-spec update 009 --priority medium --tags "feature,sprint-2"
lean-spec update 010 --priority low --tags "feature,backlog"
```

:::info Sprint Planning
Use priorities + tags to mark sprint work:
- **high** priority + **sprint-1** tag = current sprint
- **medium** priority + **sprint-2** tag = next sprint
- **low** priority + **backlog** tag = future work
:::

## Step 9: Review Bob's Spec

Before implementing the UI, review Bob's API spec to understand the contract:

```bash
lean-spec view 007
```

Look for:
- **API endpoints**: What URLs are available?
- **Data format**: What JSON structure does it return?
- **Error handling**: How are errors communicated?
- **Success criteria**: What tests were done?

If anything is unclear, talk to Bob or update the spec with clarifications.

:::tip Spec Review Checklist
When reviewing specs:
- [ ] Goal is clear and measurable
- [ ] Design decisions are documented
- [ ] Success criteria are testable
- [ ] Non-goals are explicit
- [ ] Related specs are linked
:::

Alice decides to hand off the comment system to Bob (who finished his API work):

```bash

```

Bob can see his new assignment:


:::tip Handoff Best Practice
When handing off a spec:
1. Ensure the spec is up-to-date
2. Add handoff notes in the spec (why, context)
3. Notify the new assignee
4. Make sure they understand dependencies
:::

## Team Collaboration Patterns

**Sprint Planning**:
```bash
# Tag specs for current sprint
lean-spec update 015 --tags "sprint-5,frontend"
lean-spec update 016 --tags "sprint-5,backend"

# Find sprint work
lean-spec search "tag:sprint-5"
```

**Daily Standup**:
```bash
# What's in progress?
lean-spec board

# What's blocked?
lean-spec deps <spec-number>

# Who's working on what?
lean-spec search "status:in-progress"
```

**Load Balancing**:
```bash
# Check team workload
lean-spec stats

# Find unassigned work
lean-spec search "assignee:"

# Find who has least work
lean-spec search "assignee:Alice"
lean-spec search "assignee:Bob"
```

## Team Best Practices

**1. Clear Assignments**
- Assign specs at the start of sprint
- One assignee per spec (avoid shared ownership confusion)
- Reassign explicitly if someone picks up another's work

**2. Document Handoffs**
- Add notes to spec explaining handoff context
- Update Design or Notes section with decisions made
- Ensure new assignee reviews spec and asks questions

**3. Review Before Starting**
- Always review dependent specs before starting work
- Look for API contracts, data formats, error handling
- Ask questions if anything is unclear

**4. Keep Specs Current**
- Update status as work progresses
- Document learnings and changes
- Complete specs properly (don't leave orphaned in-progress work)

**5. Use Tags Strategically**
- Sprint tags (sprint-1, sprint-2)
- Team tags (frontend-team, backend-team)
- Epic tags (checkout-epic, user-mgmt)
- Type tags (feature, bugfix, refactor)

**6. Regular Check-ins**
- Review board in standup
- Check stats weekly for health
- Identify blockers early

## Common Team Scenarios

**Parallel Work**:
```bash
# Alice and Bob work independently
lean-spec update 020 --assignee "Alice" --status in-progress
lean-spec update 021 --assignee "Bob" --status in-progress
```

**Blocked Work**:
```bash
# Check what's blocking Alice
lean-spec deps 020

# If blocked, work on something else
lean-spec search "assignee:Alice status:planned"
```

**Cross-team Dependencies**:
```bash
# Frontend depends on backend
lean-spec link 025 --depends-on 024

# Both teams can see the dependency
lean-spec deps 024  # Backend sees who needs them
lean-spec deps 025  # Frontend sees who they're waiting for
```
