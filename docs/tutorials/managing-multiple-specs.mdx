# Managing Multiple Specs

**Time**: 20 minutes  
**Goal**: Learn to manage multiple specs with dependencies, relationships, and track progress across them

## What You'll Learn

- Create and organize multiple related specs
- Set up dependencies between specs
- Use relationships to link related work
- Track progress across multiple specs
- Use the board and search effectively

## Prerequisites

- Completed [Your First Spec](./your-first-spec) and [SDD Workflow](./sdd-workflow-feature-development) tutorials
- LeanSpec installed and initialized project

## The Scenario

You're building an e-commerce platform. You need to implement:
1. User authentication (login/signup)
2. Shopping cart functionality (depends on authentication)
3. Product search (independent, but related to catalog)
4. Product catalog API (needed by search)

These specs have dependencies and relationships you need to manage.

## Step 2: Set Dependencies

The shopping cart depends on user authentication - users must be logged in to add items. Set up this dependency:

```bash
lean-spec link 004 --depends-on 003
```

**Expected output**:
```
âœ“ Updated dependencies for spec: 004-shopping-cart
  Depends on: 003-user-authentication
```

Similarly, product search needs the catalog API:

```bash
lean-spec link 005 --depends-on 006
```

:::tip Dependency Rule
Use `depends_on` when spec B **cannot start** until spec A is complete. This creates a blocking dependency.
:::

## Step 3: Set Relationships

Product search and shopping cart are related (both use products) but don't block each other. Create a relationship:

```bash
lean-spec link 005 --related 004
```

**Expected output**:
```
âœ“ Updated relationships for spec: 005-product-search
  Related: 004-shopping-cart
```

:::info Related vs Depends On
- **depends_on**: Blocking dependency (can't start B until A is done)
- **related**: Informational link (work is connected but not blocking)

Related links are **bidirectional** - when you link A to B, both specs show the relationship.
:::

## Step 5: Prioritize Work

Set priorities to guide what to work on first:

```bash
lean-spec update 003 --priority high
lean-spec update 006 --priority high
lean-spec update 004 --priority medium
lean-spec update 005 --priority low
```

Why these priorities?
- **003 (auth)**: High - needed by shopping cart
- **006 (catalog API)**: High - needed by search
- **004 (cart)**: Medium - core feature but depends on auth
- **005 (search)**: Low - nice to have, not critical path

## Step 6: Use Tags for Organization

Add tags to group related specs:

```bash
lean-spec update 003 --tags "authentication,backend,api"
lean-spec update 004 --tags "cart,frontend,feature"
lean-spec update 005 --tags "search,frontend,feature"
lean-spec update 006 --tags "api,backend,catalog"
```

Now search by tag:

```bash
lean-spec search "tag:backend"
```

**Expected output**:

:::tip Organizing with Tags
Use tags to:
- Group by layer (frontend, backend, api)
- Group by type (feature, bugfix, refactor)
- Group by epic (checkout, catalog, user-mgmt)
:::

## Dependency Patterns

**Sequential dependencies** (one depends on next):
```bash
lean-spec link 010 --depends-on 009
lean-spec link 009 --depends-on 008
```

**Multiple dependencies** (spec depends on several others):
```bash
lean-spec link 015 --depends-on 012,013,014
```

**Removing dependencies**:
```bash
lean-spec unlink 004 --depends-on 003
```

## Best Practices

**Use dependencies when:**
- Spec B truly cannot start until A is done
- There's a technical dependency (API must exist before UI)
- Work must be done in specific order

**Use relationships when:**
- Specs are related but not blocking
- Context is helpful but not required
- Working on one informs the other

**Set priorities based on:**
- Dependencies (what unblocks other work)
- Business value (what delivers most value)
- Risk (what's most uncertain, do early)

**Use tags for:**
- Quick filtering by category
- Grouping related work
- Sprint planning
- Team assignments
